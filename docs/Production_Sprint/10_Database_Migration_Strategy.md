## Problem

Over multiple development sessions, the Prisma schema (`schema.prisma`) accumulated changes that were never captured in migration files. This resulted in **schema drift** â€” the database had columns, tables, enums, and indexes not present in any migration. Running `prisma migrate reset` (which drops and re-applies only migration files) would produce a database that didn't match the schema, causing seed failures and runtime errors.

### Drift examples

| Model | Missing from migrations |
|-------|------------------------|
| `User` | `phone`, `tosAcceptedAt`, `tosVersion`, `privacyAcceptedAt`, `privacyVersion`, `consentDataAt`, `notificationsEnabled`, `pushMilestones`, `pushBadges`, `pushGrowth`, `pushProposals`, `quietHoursStart`, `quietHoursEnd` |
| `BabyMon` | `gestationalAgeAtBirth`, `dueDate`, `isGraduated`, `graduatedAt`, `siblingGroupId`, `traitsUpdatedAt` |
| Multiple | Enum changes (`MilestoneDomain`, `MilestoneStatus`, `SubscriptionTier`), new tables (`SleepLog`, `Allergy`, `MedicalTeam`, `DailyActivity`), index changes |

A hand-crafted migration (`20260624000000_add_phone_to_user`) was created to patch some columns, but it was incomplete and improperly generated. It has been **deleted**.

## Resolution

### Current state (dev)

- **7 clean migrations** in `prisma/migrations/` (all generated by `prisma migrate dev`)
- Schema drift handled by: `prisma db push --accept-data-loss`
- Database seeded with `prisma db seed`
- Shadow database URL added to `.env` for future `prisma migrate diff` commands

### Why `prisma db push` instead of a migration

`prisma migrate diff` generates SQL that includes destructive operations (e.g., `DROP CONSTRAINT` without re-creation) when comparing migrations against the schema datamodel. This is a known limitation. Rather than risk a broken migration:

1. `prisma db push` syncs the database directly from the schema (bypassing migrations)
2. This is acceptable for **development only** â€” there are no production users and no data to preserve

### Production path

Before going to production, baseline the schema into a single clean migration:

```
# 1. Create a fresh PostgreSQL database
createdb babymon_fresh

# 2. Run prisma migrate dev to generate a baseline migration
DATABASE_URL="postgresql://babymon@127.0.0.1:5432/babymon_fresh" \
  npx prisma migrate dev --name baseline

# 3. This creates one migration file with the full DDL
# 4. Test: reset the dev database and apply the new migration
# 5. Commit and deploy
```

After baselining, all future schema changes go through `prisma migrate dev` as normal incremental migrations. Never use `prisma db push` in production.

### Loss prevention

The following git rules are now in place to prevent work from being lost:

1. **Commit after every session** â€” even if the commit is messy. Squash later.
2. **Never rely on `git stash`** for work you care about. Stashes are ephemeral scratchpads. Commit instead.
3. **Run `git status --short` before closing** â€” if modified files exist and you'd be annoyed to lose them, commit.

## Related: Routine Sync Protocol Fix

### Problem
Activity names in routine templates were not unique (e.g., "Feed." appeared at both 10:30 AM and 2:00 PM). The sync protocol used activity names as identifiers, causing:
- Checked items "moving" to wrong time slots after sync
- Previously checked items being lost when new items were toggled
- The `activityToKey` reverse map was lossy for duplicates

### Fix
- **Seed data:** Made all activity names unique within each template (e.g., `"Feed. â€” mid-morning"`, `"Feed. â€” early afternoon"`). 15 duplicates across 7 templates fixed.
- **Sync protocol:** Sends the full merged list of completed activities (not just pending deltas). The backend replaces the entire `completedSteps` array, so the client must send the authoritative list.
- **Frontend guards:** Orphaned pending IDs filtered out; pending state not restored for empty/placeholder BabyMon IDs.

### Files changed
| File | Change |
|------|--------|
| `apps/api/prisma/seed-companion.ts` | Unique activity names |
| `apps/mobile/.../routine_screen.dart` | Full-merge sync, fresh serverSteps, safe pending |
| `apps/mobile/.../companion_tab.dart` | Safe pending clearing, skip restore for empty ID |
| `apps/mobile/.../milestone_tracker_screen.dart` | Orphaned pending ID filtering |

## Related: BabyMon Card Contact Number Crash

### Problem
Editing the BabyMon card's contact number crashed the app. Root cause: `getProfile` endpoint excluded `phone` from its Prisma `select`, so the field always returned `null` after reload. The save worked (backend `updateProfile` stored it correctly), but the read-back was broken.

### Fix
- Added `phone: true` to `getProfile`'s `select` in `users.service.ts`
- Added `gender: dto.gender` to `updateBabyMon`'s data payload (was silently dropped)

### Files changed
| File | Change |
|------|--------|
| `apps/api/src/users/users.service.ts` | `getProfile` now returns `phone` |
| `apps/api/src/baby-mon/baby-mon.service.ts` | `update` now saves `gender` |
