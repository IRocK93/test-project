**Date:** 2026-06-25

## Overview & Current State Assessment

The mobile app recently established a Flutter `gen-l10n` pipeline with `app_en.arb` and the `context.l10n` extension pattern. The onboarding screen (`create_baby_mon_screen.dart`) was fully localized as a pilot. However, the backend remains entirely English-only with hardcoded strings in every service, controller, and the global exception filter.

### Current state matrix

| Layer | i18n Status | Technology | Gaps |
|-------|-------------|-----------|------|
| Mobile UI copy | ✅ Partial (onboarding only) | Flutter `gen-l10n` + ARB | ~20 screens still have hardcoded strings |
| Mobile error handling | ❌ None | Raw DioException message display | Shows English backend messages directly |
| Backend API messages | ❌ None | Hardcoded English strings | All services return raw English text |
| Backend exception filter | ❌ None | Hardcoded `'Internal server error'` | No error codes for client-side mapping |
| Database/editorial | ❌ None | English seed data only | Advice cards, milestones, routines are English prose |
| Emails / Push | ❌ None | SendGrid / Firebase Admin | No template localization |
| User preference | ❌ None | No `locale` field on User | Cannot persist language choice |

---

## Guiding Principles

1. **Client owns UI copy.** The backend returns machine-readable error codes; the mobile app maps them to localized strings via `context.l10n`. This decouples release cycles.
2. **Backend localizes out-of-band communication.** Emails, push notifications, and editorial database content are localized server-side because the client is not present at send time.
3. **Fallback chain is explicit.** `requestedLocale → user.savedLocale → deviceLocale → 'en'`. Every query to translation tables uses `IN [requested, 'en']` with ordered precedence.
4. **Type safety everywhere.** Backend translation keys are TypeScript-typed via `nestjs-i18n` code generation. Mobile keys are typed via `flutter gen-l10n`.
5. **No breaking changes to existing API contracts.** Existing `message` fields remain as English developer fallbacks. New `code` fields are additive.

## 1. Backend i18n Architecture

### Technology choice: `nestjs-i18n`

**Package:** `nestjs-i18n` (v11+, compatible with NestJS 11.x already in use)  
**Justification:**
- Industry standard for NestJS with 1M+ weekly downloads
- Built-in `Accept-Language` header resolver
- Type-safe translations via generated TypeScript interfaces
- Pluralization, interpolation, and fallback locale support out of the box
- JSON format compatible with TMS (Translation Management Systems) like Crowdin, Lokalise, or Phrase

**Install:**
```bash
cd apps/api
npm install nestjs-i18n
```

### File structure

```
apps/api/src/
  i18n/
    i18n.module.ts          # Module registration & fallback config
    i18n.generated.ts       # Auto-generated by nestjs-i18n CLI (git-ignored)
    locales/
      en/
        errors.json         # API error code → English fallback message
        emails.json         # Email subject/body templates
        push.json           # Push notification titles/bodies
      es/                   # Spanish (Phase 3)
        errors.json
        emails.json
        push.json
```

### Module registration (`i18n.module.ts`)

```typescript
import { Module } from '@nestjs/common';
import {
  AcceptLanguageResolver,
  HeaderResolver,
  I18nModule,
  QueryResolver,
} from 'nestjs-i18n';
import * as path from 'path';

@Module({
  imports: [
    I18nModule.forRoot({
      fallbackLanguage: 'en',
      loaderOptions: {
        path: path.join(__dirname, '/locales/'),
        watch: process.env.NODE_ENV === 'development',
      },
      resolvers: [
        // Priority order: query string ?lang=es → Accept-Language header → fallback
        new QueryResolver(['lang', 'l', 'locale']),
        AcceptLanguageResolver,
        new HeaderResolver(['x-custom-lang']),
      ],
      typesOutputPath: path.join(
        __dirname,
        '../../src/i18n/i18n.generated.ts',
      ),
    }),
  ],
})
export class I18nConfigModule {}
```

### Type-safe usage example

```typescript
import { I18nService } from 'nestjs-i18n';
import { I18nTranslations } from '../i18n/i18n.generated';

@Injectable()
```

### Global exception filter update

The current filter returns `{ statusCode, message, code, timestamp, path }` with hardcoded English `message` values. We add an error code enum and keep `message` as a developer-facing English fallback.

**New response shape:**
```json
{
  "statusCode": 401,
  "message": "Invalid email or password",
  "code": "AUTH_INVALID_CREDENTIALS",
  "timestamp": "2026-06-25T14:32:00.000Z",
  "path": "/api/v1/auth/login"
}
```

**Mobile maps `code` to `context.l10n.errorAuthInvalidCredentials`.**

**Exception filter changes:**
```typescript
// apps/api/src/common/filters/global-exception.filter.ts
let code: ErrorCode = ErrorCode.INTERNAL_ERROR;

if (exception instanceof HttpException) {
  status = exception.getStatus();
  const res = exception.getResponse();
  message = typeof res === 'string' ? res : String(res.message || exception.message);
  code = (res as any).code || this.inferCodeFromStatus(status);
} else if (exception instanceof Prisma.PrismaClientKnownRequestError) {
  status = this.mapPrismaError(exception);
  message = 'A database error occurred';
  code = ErrorCode.DATABASE_ERROR;
}

// Log in English for ops; client localizes via `code`
this.logger.error({ err: exception, path: request.url, status, code }, message);

response.status(status).json({ statusCode: status, message, code, timestamp, path });
```

### Editorial content: translation tables

Editorial content (advice cards, milestone expectations, routine templates, vaccination schedules, screening reminders, stage content) must support per-locale translations without duplicating non-textual data.

**Pattern for every editorial entity:**

```prisma
model ExpertAdviceCard {
  id        String   @id @default(uuid())
  stageKey  String
  category  AdviceCategory
  priority  Int      @default(50)
  source    ContentSource @default(DEVELOPMENT)
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  translations ExpertAdviceCardTranslation[]
  SavedAdvice  SavedAdvice[]
  AdviceRating AdviceRating[]

  @@index([stageKey])
  @@index([category])
}

model ExpertAdviceCardTranslation {
  id      String @id @default(uuid())
  cardId  String
  locale  String // "en", "es", "fr"
  title   String
  summary String
  content String @db.Text
  tags    String[]

  card ExpertAdviceCard @relation(fields: [cardId], references: [id], onDelete: Cascade)

  @@unique([cardId, locale])
  @@index([locale])
}
```

**Apply this same pattern to:**
- `MilestoneExpectation` → `MilestoneExpectationTranslation`
- `RoutineTemplate` → `RoutineTemplateTranslation`
- `VaccinationSchedule` → `VaccinationScheduleTranslation`
- `ScreeningReminder` → `ScreeningReminderTranslation`
- `StageContent` → `StageContentTranslation`

**Query pattern with fallback:**
```typescript
const locale = i18nContext.lang; // e.g. 'es'
const cards = await prisma.expertAdviceCard.findMany({
  where: { stageKey: 'born_month_3' },
  include: {
    translations: {
      where: { locale: { in: [locale, 'en'] } },
      orderBy: { locale: 'desc' }, // 'es' sorts before 'en' alphabetically — use explicit ordering
    },
  },
});

// Flatten: pick the first translation (requested locale) or fallback to English
const localized = cards.map(card => {
  const t = card.translations.find(tr => tr.locale === locale)
         || card.translations.find(tr => tr.locale === 'en');
  return { ...card, title: t?.title, summary: t?.summary, content: t?.content };
});
```

> **Note on ordering:** Use an explicit `CASE` order or in-memory sort. `'es' < 'en'` alphabetically, so `orderBy: { locale: 'desc' }` puts 'en' first. Prefer client-side selection for clarity.

## 3. API Contract Changes

### Error response contract (additive only)

Every API error response gains a `code` field. Existing `message` fields remain unchanged as English developer fallbacks.

**Before:**
```json
{ "statusCode": 401, "message": "Invalid email or password" }
```

**After:**
```json
{ "statusCode": 401, "message": "Invalid email or password", "code": "AUTH_INVALID_CREDENTIALS" }
```

### Locale negotiation

1. **Unauthenticated requests:** `Accept-Language` header drives `nestjs-i18n` resolution.
2. **Authenticated requests:** If `user.locale` is set, it overrides the header for:
   - Email sends
   - Push notification content generation
   - Editorial content queries (advice cards, milestones)
3. **Mobile sends header:** Dio interceptor appends `Accept-Language: <device_locale>` to every request.

### New endpoint: update user locale

```
PATCH /users/me/locale
Authorization: Bearer <jwt>
Content-Type: application/json

{ "locale": "es" }
```

**Behavior:**
- Validates against supported locales whitelist (`['en', 'es']` initially)
- Updates `User.locale`
- Returns updated user profile

## 4. Mobile i18n Architecture

### Current state (keep)

- `apps/mobile/lib/l10n/app_en.arb` — template ARB file
- `apps/mobile/lib/l10n/l10n_ext.dart` — `BuildContext.l10n` extension
- `flutter gen-l10n` generates `AppLocalizations` classes

### Required additions

#### A. Error code mapping layer

Create `apps/mobile/lib/core/errors/error_mapper.dart`:

```dart
import 'package:baby_mon/l10n/l10n_ext.dart';
import 'package:flutter/material.dart';

class ErrorMapper {
  static String localize(BuildContext context, String? code) {
    final l10n = context.l10n;
    switch (code) {
      case 'AUTH_INVALID_CREDENTIALS':
        return l10n.errorAuthInvalidCredentials;
      case 'AUTH_OAUTH_REQUIRED':
        return l10n.errorAuthOauthRequired;
      case 'AUTH_ACCOUNT_DELETED':
        return l10n.errorAuthAccountDeleted;
      case 'AUTH_TOKEN_EXPIRED':
        return l10n.errorAuthTokenExpired;
      case 'RESOURCE_NOT_FOUND':
        return l10n.errorResourceNotFound;
      case 'SUBSCRIPTION_REQUIRED':
        return l10n.errorSubscriptionRequired;
      case 'DATABASE_ERROR':
      case 'INTERNAL_ERROR':
        return l10n.errorServer;
      default:
        return l10n.errorGeneric;
    }
  }
}
```

#### B. Dio interceptor for Accept-Language

Create `apps/mobile/lib/core/data/locale_interceptor.dart`:

```dart
import 'package:dio/dio.dart';
import 'package:flutter/material.dart';

class LocaleInterceptor extends Interceptor {
  final Locale? Function() getLocale;

  LocaleInterceptor(this.getLocale);

  @override
  void onRequest(RequestOptions options, RequestInterceptorHandler handler) {
    final locale = getLocale();
    if (locale != null) {
      options.headers['Accept-Language'] = locale.toLanguageTag();
    }
    super.onRequest(options, handler);
  }
}
```

Wire it into `api_client.dart`:
```dart
_dio.interceptors.add(LocaleInterceptor(() => _currentLocale));
```

#### C. Locale persistence & sync

- Store selected locale in `SharedPreferences` (key: `user_locale`)
- On app startup: read persisted locale → if different from device locale, apply it
- On login/registration: sync device locale to backend via `PATCH /users/me/locale`
- On settings language change: update local state + persist + sync to backend

#### D. ARB naming convention

Use semantic, hierarchical camelCase keys:

```
authLoginButton
authErrorInvalidCredentials
authErrorAccountDeleted
onboardingSplashTitle
onboardingSplashSubtitle
onboardingNameHint
errorGeneric
errorServer
errorConnection
```

**Avoid:** screen-scoped prefixes like `loginScreen_emailLabel` — keys should describe intent, not location, so they can be reused across screens.

### Email localization

```typescript
// Backend service
async sendVerificationEmail(email: string, token: string, locale: string) {
  const templateId = locale === 'es'
    ? this.config.sendgridTemplateEsVerify
    : this.config.sendgridTemplateEnVerify;

  await this.sendgrid.send({
    to: email,
    templateId,
    dynamicTemplateData: { verificationUrl: `${this.config.frontendUrl}/verify?token=${token}` },
  });
}
```

## 9. Edge Cases, Risks & Mitigation

| Risk | Impact | Mitigation |
|------|--------|-----------|
| **Incomplete Spanish translations** | UI shows English fallback | Mobile: `flutter gen-l10n` auto-fallbacks to template locale. Backend: `IN [requested, 'en']` query pattern. |
| **Backend message shown before mobile update** | Old app versions show raw English | Keep `message` as English fallback in API. Old clients ignore `code`; new clients use `code`. Breaking change: none. |
| **Push notification locale drift** | User changed language but backend hasn't synced | Backend always uses `user.locale` (DB) for push, not ephemeral headers. Mobile syncs on every app foreground. |
| **Email template missing for locale** | SendGrid fails or sends blank | Fallback template ID chain: `es` → `en`. Always have English templates. |
| **Long translations break UI layout** | German/Spanish text overflows | Widget tests with `Locale('es')`. Use `AutoSizeText` or `Flexible` where appropriate. Design for +30% text expansion. |
| **Translator modifies ARB and breaks syntax** | `flutter gen-l10n` fails | CI step runs `flutter gen-l10n` on every PR. Fails build if ARB is invalid. |
| **Performance: translation table joins slow queries** | Advice feed loads slowly | Add `@unique([cardId, locale])` indexes. Use `include` with `where: { locale }` to limit rows. Cache hot content in Redis if needed. |
| **RTL languages (Arabic, Hebrew)** | Layout breaks completely | Phase 5 audit. Use `Directionality.of(context)` instead of hardcoded `TextDirection.ltr`. Test with `Locale('ar')`. |
| **Error code enum divergence** | Mobile has code `AUTH_BAD_CREDS`, backend sends `AUTH_INVALID_CREDENTIALS` | Single source of truth: backend enum exported as JSON, mobile codegen generates Dart enum from it. Or shared schema repo. |
